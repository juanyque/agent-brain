# cleanup_ds_store.py

## Purpose
- Remove `.DS_Store` files generated by macOS from visible vault content.
- `.DS_Store` files are noise. They also prevent directories from being detected as empty by subsequent cleanup passes (e.g. `vault_setup.py`'s `cleanup_empty_dirs_recursively`), which leaves stale empty trees in the vault.

## Safety model
- Dry-run by default.
- Top-level dotfile dirs (`.git/`, `.obsidian/`, `.WIP_<timestamp>/`, etc.) are skipped entirely — they may contain their own `.DS_Store` files that must not be touched.
- Symlinks are never followed or removed.
- Only files whose basename matches `NOISE_FILE_NAMES` (currently `[".DS_Store"]`) are removed. The list is a constant at the top of the script; extend it in place when a new universal noise pattern emerges.

## Usage

### Dry-run
```bash
python3 ~/.agents/skills/obsidian/scripts/cleanup_ds_store.py --vault-root /path/to/vault
```

### Apply
```bash
python3 ~/.agents/skills/obsidian/scripts/cleanup_ds_store.py --vault-root /path/to/vault --apply
```

## When this runs
- **Automatically inside `vault_setup.py`** before `cleanup_empty_dirs_recursively`, so directories that hold only `.DS_Store` are correctly detected as empty afterwards. Gated by `--skip-full-reorder`: when the user opts out of vault reordering, this sweep is also skipped.
- **As a safe pre-check during `/obsidian init` maintenance mode**, run automatically without per-action confirmation. Removing `.DS_Store` does not destroy information, so it is one of the few maintenance actions that does not require an `AskUserQuestion` gate.
- **On demand**, invoked manually by the user or agent at any time.

## Known limitations
- Only handles macOS noise (`.DS_Store`). Other OS/filesystem noise patterns (`Thumbs.db` on Windows, `@eaDir` on Synology) are not currently removed.
- To extend coverage, append to `NOISE_FILE_NAMES` in the source. If the list grows beyond ~3 entries or needs per-vault customization, consider renaming the script to `cleanup_noise_files.py` and exposing `--pattern` on the CLI.

## Logging
- Every run prints to the console. The script does not maintain a `.log` file; output is captured by the caller (e.g. `vault_setup.py` includes it in its own log) or by the user when run interactively.
