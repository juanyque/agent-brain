# Review evidence rules

Use this rule when creating, updating, archiving, or referencing evidence and reports that support performance and review cycles. The model separates a **continuous evidence store** (self-contained atomic notes, append-only) from **on-demand reports** (brag, feedback, complaint — generated by filtering the store, then curated by the human).

The procedural guides live in `TASK_TYPES/evidence-management.common.md` (the store), `TASK_TYPES/brag-report.common.md`, `TASK_TYPES/feedback-report.common.md`, and `TASK_TYPES/complaint-report.common.md` (the reports). This rule defines the conventions those guides rely on.

## Scope and relationship to other layers

- **Evidence store** lives under `WIP/evidence/`. Each item is an atomic note with structured frontmatter and a self-contained prose body. Evidence is append-only: it accumulates continuously and is not scoped to a particular review cycle or year.
- **Reports** are generated on demand by filtering the store (by kind, date range, topic, or person). A report is a transient, curated artifact: generated as a draft, reviewed and enriched by the human, shared when needed, then archived. Reports live in `WIP/`.
- **Archive**: when a report's cycle closes (review submitted, complaint resolved, fiscal period over), it moves to `ARCHIVED/Reviews/` via `git mv`. The evidence store is never archived — it is the permanent raw material; reports are the point-in-time output.
- People referenced by evidence and reports live in `MEMORY/People/`. Evidence notes link to people notes via the `people:` frontmatter field; people notes may have an optional `## Review history` section linking back.

## Three-layer model

```
CAPTURE               STORE                    REPORT
(daily note)          (WIP/evidence/)          (WIP/ — on demand)
[[BRAG]]: stub   →    atomic note         →    brag report (filtered by kind + date range)
[[FEEDBACK]]: stub →  YYYYMMDD-HHMMSS-slug →   feedback report (filtered by kind + cycle/person)
[[COMPLAIN]]: stub →  .md                 →    complaint report (filtered by kind + topic)
```

1. **Capture**: daily note action categories (`[[BRAG]]`, `[[FEEDBACK]]`, `[[COMPLAIN]]`) hold one-line stubs. The human writes what happened; the detail is deferred to harvest.
2. **Store**: the harvest process (see `TASK_TYPES/evidence-management.common.md`) reads unlinked stubs, creates self-contained atomic notes, and rewrites the daily line with a wikilink to the new note. Historical evidence can be backfilled directly into the store.
3. **Report**: reports are generated by querying the store's frontmatter (scripts — deterministic, no LLM needed for filtering), then assembling a coherent narrative (LLM), then curating impact and reflection (human). The report is a draft until curated.

## Evidence note schema

Template: `TEMPLATES/TEMPLATE.evidence-note.common.md`.

| Field | Type | Description |
|---|---|---|
| `date` | `YYYY-MM-DD` | Date the event occurred. |
| `kinds` | list | `brag`, `feedback`, `complaint`, `incident`, `learning` (open enum; multiple allowed). |
| `topic` | string | Project/theme slug. |
| `people` | list | Wikilinks to `MEMORY/People/` notes. |
| `source` | string or null | `[[daily]]`, `![[attachment]]`, URL, or null. |
| `backfilled` | bool | True if ingested directly (not via daily harvest). |
| `sensitive` | bool | True for complaints, interpersonal incidents, or content naming people negatively. |
| `tags` | list | Always `evidence`; add `sensitive` when applicable. |

The note **body** is the detail: a self-contained prose account. Transcribe attachment content so the note is searchable without the binary.

## Naming conventions

Follow `RULES-FILE-NAMING.common.md`. Specific patterns:

- **Evidence note**: `YYYYMMDD-HHMMSS-<slug>.md` under `WIP/evidence/`. The date is the event date; the timestamp includes seconds to disambiguate same-minute captures. Example: `20260715-103025-billing-rest-api-shipped.md`.
- **Report**: `report-<kind>-YYYYMMDD-<slug>.md` under `WIP/`. `<kind>` is `brag`, `feedback`, or `complaint`. The date is the generation date. Example: `report-brag-20260701-fy25-h2.md`, `report-complaint-20260720-arch-review-conflict.md`.
- **Attachments**: stored in `WIP/evidence/ATTACHMENTS/`, named date-first: `YYYYMMDD - <source> - <topic>.<ext>`.

Basenames must stay unique vault-wide (Obsidian resolves wikilinks by basename).

## Lifecycle

### Evidence (the store)

```
daily stub  →  atomic note in WIP/evidence/  (permanent)
```

Evidence notes are **append-only and permanent**. They do not move to `ARCHIVED/`. If the factual understanding of an item changes, add a dated addendum to the body rather than rewriting the original.

### Reports (the transient layer)

```
generated draft (WIP/)  →  curated (WIP/)  →  submitted/delivered  →  ARCHIVED/Reviews/ (cycle closed)
```

1. **Generate**: query the evidence store by filter, assemble a draft narrative. The draft is mechanical output.
2. **Curate**: the human adds impact, reflection, and narrative coherence. This step is what turns a filtered list into a useful document.
3. **Submit**: update the report's `status` frontmatter to `submitted` (or `delivered`, `filed` — depending on the report kind).
4. **Archive**: after the cycle is fully closed, move the report to `ARCHIVED/Reviews/` with `git mv`. Keep the same basename. Reports tagged `sensitive` (complaint reports) stay in `WIP/` unless the user explicitly decides to archive them, because retention may be governed by HR policy.

## Daily note integration

- The daily template includes three review-evidence action categories (see `TEMPLATES/TEMPLATE.daily-note.common.md`):
  - `* [[BRAG]]:` — accomplishments, shipped work, recognitions.
  - `* [[FEEDBACK]]:` — feedback given or received.
  - `* [[COMPLAIN]]:` — incidents relevant to a complaint or interpersonal conflict.
- Each category holds one-line stubs. Before harvest, a stub is plain text. After harvest, the stub is rewritten with a wikilink to the atomic note:
  ```
  * [[BRAG]]:
    * [[20260715-103025-billing-rest-api-shipped]] Shipped the new REST API for the billing service
  ```
- These are the backward-looking counterpart of `* [[OBJECTIVES]]:` (forward-looking goals). An entry can feed both: record objective progress under `[[OBJECTIVES]]`, add a stub under `[[BRAG]]` if the accomplishment framing adds value.
- At end-of-day cleanup, empty categories are removed like any other (see `RULES-DAILY-NOTES.common.md`).

## Attachments and evidence

Screenshots, exported emails, and other binary evidence are stored in `WIP/evidence/ATTACHMENTS/`.

- The evidence note's **body transcribes** the relevant content of the attachment in prose. The transcription makes the note searchable and readable without opening the file; the binary is the proof kept for when it is requested.
- This applies to all evidence kinds. For brag items, the attachment might be a screenshot of a metric or a peer message on Slack. For complaints, it might be a screenshot of the incident. The transcription principle is the same.
- Attachment naming follows `RULES-FILE-NAMING.common.md` (date-first when the date has value). Example: `20260315 - Slack - jane-arch-review-tone.png`.

When a report moves to `ARCHIVED/Reviews/`, its own attachments (if any, added during curation) move with it. Evidence note attachments in `WIP/evidence/ATTACHMENTS/` do not move — they belong to the permanent store.

## Sensitivity

- **Default**: brag and feedback evidence is normal review content. No special handling beyond the standard git workflow.
- **Sensitive flag**: evidence notes with `sensitive: true` (complaints, interpersonal incidents, content naming people negatively) carry `tags: [evidence, sensitive]`. The `sensitive` tag signals:
  - Maintenance jobs must not move, rename, archive, or bulk-edit these notes without explicit user confirmation.
  - The note should not be quoted or surfaced in generated summaries, reports, or agent output without the user explicitly asking for it.
- **Retention**: HR policies sometimes require destroying drafts after a complaint is formalized. Git preserves history indefinitely; if destruction is required, that is a deliberate `git filter-repo` operation the user must run explicitly. The operating model never destroys content automatically.
- **Named individuals**: negative feedback or complaint content names real people. Only record what is factual, observed, and necessary. Separate facts (what happened, when, where) from interpretations (what it means). The evidence note body enforces this by being a factual account; the report's curation step is where interpretation belongs.
- **People slugs**: use `MEMORY/People/` wikilinks in the `people:` field. The person's real name lives in their People note, not scattered in evidence bodies. This keeps evidence notes slightly less identifying if partially exposed.

## Relationship to `WIP/OBJECTIVES.md`

- `WIP/OBJECTIVES.md` is forward-looking: what the user is working toward, with a "how to evidence" line per objective.
- The evidence store is backward-looking: what happened.
- The bridge is the daily note. `[[OBJECTIVES]]:` captures forward progress; `[[BRAG]]:` / `[[FEEDBACK]]:` / `[[COMPLAIN]]:` capture backward evidence. A single daily entry can feed both.

## Report aging (maintenance)

- Reports live in `WIP/` while being curated or while the cycle is open.
- A maintenance job (part of the obsidian skill's `maintenance_scheduler.py`) periodically reviews aged reports in `WIP/` and proposes moving them to `ARCHIVED/Reviews/` once the cycle has closed and the report is no longer actively referenced.
- Evidence notes are never proposed for archival — they are permanent store.
